01

Overview & Rationale

The floating widget is a non-intrusive alternative to the inline iframe embed. Unlike the iframe approach which requires a designated container and can disrupt existing page layouts, the floating widget overlays the page without affecting any DOM elements.

📱 Mobile-First Priority: The majority of traffic comes through mobile devices. This guideline prioritizes mobile experiences — all components are designed touch-first with mobile viewport constraints in mind. Desktop is the adaptation, not the default.

Key benefits

  • Zero layout impact — Uses fixed positioning, doesn't push or shift any page elements
  • Familiar pattern — Users recognize the floating chat button from Intercom, Drift, Zendesk, etc.
  • Progressive disclosure — Minimal footprint when collapsed (just the FAB), full experience when expanded
  • Easy integration — Single script tag, no container needed
  • Touch-optimized — 56px FAB meets touch target guidelines, full-width panel on mobile

When to use

Use floating widget when: Client sites have complex layouts, dynamic content, or the inline iframe would break their design. Also ideal for SPAs where container injection is problematic.

Use inline widget when: Client wants the chat deeply integrated into a specific page section (e.g., dedicated contact page) or needs the two-phase expand behavior.

05

Widget states

The floating widget maintains the same conversational states as the inline widget, adapted for the floating context.

STATE 01
Collapsed (FAB only)

Default state. Only the FAB is visible. Panel is hidden. Optional notification badge may be shown.

Entry: Page load, or user closes panel
Exit: User clicks FAB → Open panel
STATE 02
Expanded — Idle

Panel is open showing the greeting message and quick reply pills. Input is empty and ready for user.

Entry: User clicks FAB to open
Exit: User types message or closes panel
STATE 03
Expanded — Composing

User is typing. Send button becomes active (#000). Quick reply pills may hide to save space.

Entry: User focuses input and types
Exit: User sends message or clears input
STATE 04
Expanded — Agent typing

User message sent. Agent shows typing indicator (three bouncing dots). Input disabled during this time.

Entry: User sends message
Exit: Agent response received
STATE 05
Expanded — Conversation active

Ongoing back-and-forth conversation. Messages stack in scrollable area. Quick replies hidden.

Entry: After first exchange
Exit: Session end, error, or user closes
STATE 06
Contact collected

Agent has collected visitor's contact info (email/phone/name). Show confirmation badge and continue conversation.

Show green confirmation: "✓ Contact saved"
Conversation continues normally after.
STATE 07
Session complete

Conversation goal achieved. Show success state with summary. Option to start new conversation.

Visual: Success banner with checkmark
Action: "Start new conversation" link
STATE 08
Error state

Network error or API failure. Show error message with retry option. Don't lose conversation history.

Visual: Red error banner
Message: "Something went wrong. Tap to retry."
STATE 09
Out of scope

User asks something outside the agent's knowledge. Show amber warning and redirect to appropriate action.

Visual: Amber info banner
Message: Varies based on context

07

Positioning & responsive

The widget uses fixed positioning and adapts to different viewport sizes. Mobile is the primary design target — desktop is the adaptation.

📱 Mobile-first sizing: Design for 320px viewport first, then scale up. The panel should feel native on mobile, not like a shrunk desktop widget.

Position options

Bottom-right (default) Most common pattern, users expect it here
Bottom-left Alternative when right side is occupied

Offset specifications

ViewportFAB offsetPanel behavior
Mobile (<480px)16px from edgeFull-screen takeover (100vw × 100vh)
Desktop (≥480px)20px from edgeFloating panel (380px width)

Mobile adaptations

Full-screen takeover: On mobile (<480px), the chat takes 100% of the viewport — not a floating panel. This creates an immersive, app-like experience.

FAB behavior: FAB hides when full-screen chat is open, reappears when closed.

Back button: Primary dismiss action (familiar mobile pattern), plus optional X button.

Input: Panel accounts for virtual keyboard (visualViewport API). Header hides when keyboard opens; messages shrink to fill remaining space and input stays pinned just above the keyboard — last message stays in view.

Safe areas: Respect env(safe-area-inset-*) for notched devices.

Z-index hierarchy

/* Ensure widget floats above page content */ .contextus-widget { --z-fab: 2147483646; /* Max minus 1 */ --z-panel: 2147483645; /* Below FAB */ --z-backdrop: 2147483644; /* Optional overlay */ }

Floating bubbles appearance ✦

An alternative FAB appearance mode where quick-reply pills "pop off the page" as standalone floating bubble cards stacked above the FAB — visible before the panel ever opens. Tapping a bubble opens the chat and auto-sends that message, giving visitors a frictionless one-tap entry into a contextual conversation.

✦ New in v1.1: Floating bubbles is an opt-in appearance variant. Default behavior (pills inside the panel) is unchanged. Enable with data-appearance="bubbles" on the script tag.

How it works

1. Page load — Three bubble pills appear above the FAB with a staggered entrance animation (bottom pill first, each 110ms apart).

2. Pill tap — The bubbles animate out toward the FAB, the panel opens, and the pill text is auto-sent as a visitor message. The conversation starts immediately — no typing required.

3. FAB tap (direct) — Panel opens normally with no pre-sent message. Bubbles animate out.

4. Panel closed without conversation — Bubbles re-appear with entrance animation. Visitor gets another chance.

5. Panel closed after conversation started — Bubbles stay hidden. Conversation context is preserved.

Mobile behavior (dark theme)

Same layout, dark palette: Bubble pills switch to the dark theme (#2a2a2a background, #ddd text, lighter border) to match the mobile full-screen dark chat panel.

Pulse rings still animate: The FAB pulse rings continue to draw attention even with bubbles visible — they complement each other rather than conflict.

Tap target: Each pill is at minimum 44px tall to meet iOS/Android touch guidelines. Long pill text wraps rather than truncating.

Panel open on mobile: Bubbles animate out, full-screen takeover panel slides up as usual.

Source of pill content

Same pipeline as standard pills — bubbles are populated from the same pills[] array returned by /api/session. Before the session resolves, fallback pills are shown. Once the session returns knowledge-base-specific suggestions, they replace the fallbacks in-place (with a subtle fade transition if bubbles are still visible).

Always exactly 3 pills — the backend guarantees 3 entries. The UI should always render all 3.
